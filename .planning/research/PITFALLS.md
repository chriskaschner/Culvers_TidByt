# Pitfalls Research

**Domain:** Visual polish, design system consolidation, cone rendering upgrade, UX fixes on existing vanilla CSS/JS site
**Researched:** 2026-03-13
**Confidence:** HIGH -- all pitfalls verified against actual codebase files, line numbers, and counts

## Critical Pitfalls

### Pitfall 1: Token Rename Instead of Token Add

**What goes wrong:**
Renaming existing CSS custom properties (e.g., `--brand` to `--color-brand-primary`) to follow a "proper" design system naming convention. Every rule, HTML file, and JS innerHTML string referencing the old name silently breaks -- elements revert to browser defaults with zero console errors.

**Why it happens:**
Design system guides recommend semantic naming hierarchies (primitive -> semantic -> component). The temptation is to rename all 37 existing tokens to match. But with 15+ HTML pages, 7 JS files generating innerHTML, and a 3833-line style.css all consuming these tokens, a rename is effectively a rewrite.

**How to avoid:**
1. Add new tokens alongside existing ones. Never rename, never remove.
2. If a naming scheme change is desired later, add aliases: `--color-brand-primary: var(--brand)`.
3. Before modifying ANY token, grep all `.js`, `.html`, and `.css` files for that exact token name.
4. New tokens for rarity unification and semantic state should use new names that do not collide with existing tokens.

**Warning signs:**
Browser renders with missing colors -- text becomes black, backgrounds become white/transparent. No console errors. The 37 token names are load-bearing infrastructure at this point.

**Phase to address:**
Design token consolidation phase. Establish additive-only policy as phase rule #1.

### Pitfall 2: CSS-Only Class Rename Without JS innerHTML Audit

**What goes wrong:**
Renaming CSS classes in `style.css` and HTML files but missing the same class names embedded in JS string concatenation / innerHTML in shared-nav.js, compare-page.js, today-page.js, and fun-page.js.

**Why it happens:**
CSS refactoring tools and IDE find-replace cover `.css` and `.html` files. JS innerHTML strings with class names are invisible to these tools. This project generates significant HTML via JS string concatenation:
- `shared-nav.js`: Nav items, store indicator, entire footer
- `compare-page.js`: Picker done button (`class='btn-primary compare-picker-done'` at line 600), add-hint element with `style.cssText` at line 797
- `today-page.js`: Dispatches `sharednav:storechange` events
- `fun-page.js`: Quiz card rendering
- `map.html`: Inline `<script>` blocks with template literals for store cards (line 1034)

**How to avoid:**
1. Before renaming ANY CSS class, `grep -r "className" docs/*.js docs/*.html` for that exact class name.
2. Update all references in the same commit.
3. Run Playwright tests after every class rename -- 40 browser specs will catch visual regressions.

**Warning signs:**
Buttons appear unstyled (no background, no border-radius, wrong text color) at 375px width. Elements generated by JS look different from elements in static HTML.

**Phase to address:**
Card/button unification phase. Every class rename needs a cross-file grep step.

### Pitfall 3: Competing `.btn-primary` Definitions Across Files

**What goes wrong:**
Unifying `.btn-primary` into a single consistent style in `style.css` while forgetting that `fun.html` (line 126) has a COMPLETE redefinition of `.btn-primary` in a `<style>` block, and `updates.html` (line 75) has a scoped override of `.cal-action a.btn-primary`. These page-scoped definitions have DIFFERENT padding values:

```
style.css:   padding: var(--space-3) var(--space-6)   /* 0.75rem 2rem */
fun.html:    padding: var(--space-2) 1.25rem           /* 0.5rem 1.25rem */
updates.html: padding: var(--space-2) 1.25rem          /* 0.5rem 1.25rem */
compare.html: inline style="padding:0.5rem 1.25rem"
```

**Why it happens:**
Each page's `.btn-primary` was implemented at a different time with slightly different sizing needs. The page `<style>` blocks override the shared definition and will continue to override after "unification" -- making the unification silently incomplete.

**How to avoid:**
1. Before declaring button unification "done," grep all HTML files for `<style>` blocks containing `btn-primary`, `btn-secondary`, or `.btn`.
2. Remove or reconcile EVERY page-scoped override -- not just the ones in `style.css`.
3. If pages genuinely need different button sizes, use modifier classes (`.btn-primary--compact`) rather than page-scoped redefinitions.
4. Verify that the unified definition works at 375px width on EVERY page that uses it.

**Warning signs:**
Buttons look correct on Today but different on Fun or Updates. Visual inconsistency that is the exact thing this milestone is supposed to fix.

**Phase to address:**
Card/button unification phase. Must audit ALL `<style>` blocks in HTML files as the first step.

### Pitfall 4: 97 Hardcoded `border-radius` Values Undermining Unification

**What goes wrong:**
Adding `--radius` tokens to the design system but only converting the `.card` and `.btn` classes, while 97 `border-radius` declarations in style.css use 12 different hardcoded values (2px, 3px, 4px, 6px, 8px, 10px, 12px, 14px, 16px, 20px, 50%, 999px). Of these 97, only ~15 use design tokens (`var(--radius)`, `var(--radius-lg)`, `var(--radius-full)`). The other 82 remain hardcoded.

**Why it happens:**
Card/button unification feels complete when `.card` and `.btn-primary` use tokens. But dozens of related elements (chips, badges, toasts, search inputs, skeleton loaders, prediction bars, signal cards) each have their own hardcoded radius. The system LOOKS unified at the component level but is actually fragmented.

**How to avoid:**
1. Establish a radius scale: `--radius-sm: 3px`, `--radius: 0.5rem`, `--radius-md: 8px`, `--radius-lg: 0.75rem`, `--radius-full: 999px`.
2. Convert ALL 97 border-radius values to use the nearest token in the scale.
3. Do this as a single pass -- not incrementally per-component -- so the full count is known upfront.
4. Use `grep -c 'border-radius:' docs/style.css` before and after to verify coverage.

**Warning signs:**
After "unification," a grep for hardcoded `border-radius` values still returns 80+ results.

**Phase to address:**
Design token consolidation phase, as a dedicated sub-task after color tokens.

### Pitfall 5: Partial Golden Baseline Regeneration

**What goes wrong:**
Updating only Hero tier golden baselines after changing a shared rendering function (e.g., color lookup, topping resolution, scoop geometry). The change "only affects the Hero tier" -- but shared functions like `getFlavorProfile()`, `resolveHDToppingSlots()`, and color lookup constants affect all 4 tiers (Mini 9x11, HD 18x21, Premium 24x28, Hero 36x42).

**Why it happens:**
94 flavors x 4 tiers = 376 pixelmatch baselines, all at zero tolerance (seeded PRNG means any pixel diff is real). Developers regenerate the tier they were working on but forget that shared functions cascade to all tiers.

**How to avoid:**
1. Always run `UPDATE_GOLDENS=1 npx vitest run golden-baselines.test.js` (regenerates ALL tiers).
2. Then verify with `npx vitest run golden-baselines.test.js` (zero-tolerance check).
3. Then regenerate hero PNGs: `node scripts/generate-hero-cones.mjs` (production assets).
4. Then bump SW cache version in `docs/sw.js`.
5. All four steps in the same commit. Partial regeneration is guaranteed to fail CI.

**Warning signs:**
Hundreds of pixelmatch test failures in CI, concentrated in tiers that were not explicitly regenerated.

**Phase to address:**
Cone rendering upgrade phase. Must be its own isolated phase -- no other CSS/JS changes in flight during PNG regeneration.

### Pitfall 6: Cone Renderer Changes Interleaved with CSS/JS Work

**What goes wrong:**
Making cone renderer changes (which regenerate 470+ binary files: 94 hero PNGs + 376 golden baselines) in the middle of CSS/JS refactoring phases. Git cannot merge binary files -- every conflict requires full regeneration.

**Why it happens:**
Cone work is visually rewarding and developers want to do it first. But a commit touching 470 binary files creates an enormous diff that makes parallel work impossible. If CSS/JS work is happening in parallel or even in adjacent commits, the binary diffs make any conflict resolution painful.

**How to avoid:**
1. Complete ALL CSS/JS refactoring (tokens, buttons, inline style elimination, Compare UX fix) BEFORE starting cone renderer work.
2. Cone renderer changes go in their own dedicated phase, committed as a single batch.
3. No other changes should be in-flight when the cone regeneration commit is created.
4. Phase ordering: tokens -> buttons/cards -> inline style cleanup -> Compare UX -> test cleanup -> cone rendering (LAST).

**Warning signs:**
Git merge conflicts on `.png` files in `docs/assets/` or `worker/test/fixtures/goldens/`.

**Phase to address:**
Roadmap structure itself. Cone phase must be ordered last among functional changes.

### Pitfall 7: Compare Page Waiting Forever for Geolocation

**What goes wrong:**
Making Compare's `loadAndRender()` await SharedNav's geolocation promise with no timeout. SharedNav has an 8-second timeout. If geolocation fails or is slow, Compare shows a loading spinner for 8+ seconds before falling back.

**Why it happens:**
The fix seems simple: "just wait for SharedNav to resolve." But SharedNav's geolocation has multiple failure modes: IP geolocation API down, browser geolocation denied, network timeout, no stores in manifest. The `sharednav:storechange` event may fire before or after Compare's JS initializes, creating a race condition.

**How to avoid:**
1. Race the `sharednav:storechange` event against a 3-second timeout.
2. If the event fires within 3 seconds, use the resolved store.
3. If timeout fires first, show the empty state with "Add stores" CTA.
4. Clean up the event listener after either path resolves to prevent double-render.
5. Test with geolocation API blocked (Playwright route returning 500 for `/api/v1/geolocate`).

**Warning signs:**
First-time visitors on Compare see an indefinite spinner. Playwright test for Compare page hangs waiting for content to appear.

**Phase to address:**
Compare first-load UX phase. Must handle race condition between SharedNav geolocation and Compare initialization.

### Pitfall 8: Rarity Color Unification Breaks Map Popups

**What goes wrong:**
Unifying the two incompatible rarity color scales but only testing Today/Compare pages, not map popups. The map page renders rarity chips inside Leaflet popups using `.popup-rarity-chip` classes, which have a completely different color palette from `.rarity-badge-*` classes used elsewhere:

```
Map popups (.popup-rarity-chip):   ultra-rare=#7b1fa2 (purple), rare=#1565c0 (blue), uncommon=#2e7d32, common=#e8f5e9, staple=#f5f5f5
Today/Compare (.rarity-badge-*):   ultra-rare=#fce4ec (pink),  rare=#fff3e0 (orange), uncommon=#e3f2fd, common=#f5f5f5, staple=#e8f5e9
```

**Why it happens:**
These two scales were implemented at different times for different visual contexts (saturated chips on dark map popups vs pastel badges on white cards). A unification that picks one palette will look wrong in the other context.

**How to avoid:**
1. Define shared `--rarity-ultra-rare`, `--rarity-rare`, `--rarity-uncommon`, `--rarity-common`, `--rarity-staple` tokens in `:root`.
2. Update BOTH `.rarity-badge-*` AND `.popup-rarity-chip.rarity-*` to use the same tokens.
3. If the visual contexts truly need different palettes, use CSS scoping: `--rarity-rare` in `:root` for cards, overridden inside `.leaflet-popup-content` for map popups.
4. Test both the Compare page and the Map page popups after the change.

**Warning signs:**
Rarity chips in map popups become unreadable (low contrast) or visually jarring (wrong palette for dark/light context).

**Phase to address:**
Design token consolidation phase. Rarity tokens need a unified approach with potential context-specific overrides.

### Pitfall 9: JS Inline Style Mutations Invisible to CSS Auditing

**What goes wrong:**
Declaring "all inline styles eliminated" after cleaning up `style=""` attributes in HTML files, while JS files still set inline styles programmatically. Current inventory of JS inline style mutations:

```
today-page.js:   todayCard.style.borderLeftColor = color;         (line 359)
today-page.js:   todayCard.style.borderLeftColor = '#bdbdbd';     (line 369)
today-page.js:   todayCard.style.borderLeftColor = '#e0e0e0';     (line 378)
today-page.js:   todayFlavor.style.color = '#999';                (line 382)
today-page.js:   todayFlavor.style.color = '';                    (lines 364, 373)
updates-page.js: statusEl.style.color = '#2e7d32';                (line 82)
updates-page.js: statusEl.style.color = '#c62828';                (lines 86, 91)
compare-page.js: hint.style.cssText = 'border:2px dashed #ccc;...' (line 797)
compare-page.js: item.style.display = '' / 'none';               (lines 660, 671)
shared-nav.js:   item.style.display = '' / 'none';               (lines 407, 418)
```

**Why it happens:**
`grep -r 'style="' docs/*.html` catches attribute-based inline styles but misses programmatic `element.style.property = value` in JS. The JS mutations are particularly insidious because they use hardcoded hex values that bypass the design token system entirely.

**How to avoid:**
1. Audit JS files with `grep -r 'style\.' docs/*.js` (not just `style="`).
2. Replace `element.style.borderLeftColor = color` with class toggling: `element.classList.toggle('card--confirmed')`.
3. Replace hardcoded color values in JS (`'#bdbdbd'`, `'#2e7d32'`, `'#c62828'`) with CSS classes that reference tokens.
4. `style.display = 'none'` is acceptable (standard show/hide pattern), but `style.color` and `style.borderLeftColor` with hardcoded values are not.

**Warning signs:**
`grep -rc 'style="' docs/*.html` shows zero, but visual inspection reveals hardcoded colors on dynamic elements.

**Phase to address:**
Card/button unification and token consolidation phases should both address this. JS inline style mutations are part of the "eliminate hardcoded colors" scope, not just the "eliminate inline styles in HTML" scope.

## Technical Debt Patterns

Shortcuts that seem reasonable but create long-term problems.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Page-scoped `<style>` blocks for new CSS | Fast iteration, no style.css merge concerns | CSS rules split between style.css and 5+ HTML files (fun.html, quiz.html, updates.html, group.html, flavor-audit.html, masterlock-audit.html have `<style>` blocks) | Never in v1.5 -- consolidation milestone should reduce, not increase, CSS fragmentation |
| Inline `style=""` in JS innerHTML | Quick visual fix in JS-generated elements | Bypasses design token system entirely, invisible to CSS auditing | Never -- the milestone's purpose is eliminating these |
| Adding rarity tokens without removing old hardcoded values | New tokens work, old rules still render | Two parallel systems (tokens + hardcoded hex) coexist, negating consolidation | Never -- replace in the same commit that adds the token |
| Skipping SW cache bump after CSS/PNG changes | Fewer commits, faster iteration | Returning users see stale styles/images from SW cache | Never -- SW bump required after any change to cached assets |
| Regenerating only changed golden tiers | Faster test cycle during development | Forgotten tiers drift from actual rendering, CI fails unpredictably | During local development ONLY -- full regeneration required before commit |
| Converting only `.card` and `.btn` border-radius to tokens | Quick win for the main components | 82+ remaining hardcoded border-radius values create false sense of "done" | Never in a consolidation milestone -- partial conversion is worse than none |

## Integration Gotchas

Common mistakes when connecting the pieces of this specific system.

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| SharedNav + Compare page | Assuming `sharednav:storechange` fires after Compare JS initializes | Race event against timeout; handle both orderings |
| SharedNav + All pages | Changing store picker behavior for Compare and breaking it on Today/Map/Fun | SharedNav runs on ALL 15 pages -- test store picker changes on at least 3 pages, not just the one being fixed |
| Service worker + Playwright tests | Forgetting SW mock -- SW intercepts `page.route()` handlers | Every new Playwright test must either mock sw.js (route `**/sw.js` to no-op) or stub `navigator.serviceWorker.register` |
| cone-renderer.js + flavor-colors.js | Client-side FALLBACK_FLAVOR_ALIASES getting out of sync with server-side FLAVOR_ALIASES | After any alias change, verify both files match -- no CI gate for this sync currently |
| Design tokens + JS innerHTML | Adding new CSS classes that use tokens, but JS innerHTML strings still use old classes or inline styles | Grep all `.js` files for class names and `style=` after every CSS refactor |
| style.css + page `<style>` blocks | Defining a rule in style.css that is overridden by a page `<style>` block in fun.html, quiz.html, updates.html, group.html, or masterlock-audit.html | Check page-level `<style>` blocks for conflicts when modifying shared rules |
| style.css + JS `.style.*` mutations | Consolidating colors into tokens in CSS while JS still sets `element.style.color = '#2e7d32'` directly | Grep `style\.` in JS files, not just `style="` in HTML files |
| map.html inline `<script>` + style.css | Map page builds store cards via template literals in inline script (line 1034) using `.card.card--map-store`; CSS changes to these classes must also verify the JS-generated HTML | Test map page store card rendering after any `.card` class changes |

## Performance Traps

Patterns that work at small scale but fail as usage grows.

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Adding design tokens without removing hardcoded values | style.css grows without shrinking; currently 3833 lines with 216 hardcoded hex values and 82 hardcoded border-radius values remaining | Each new token must be accompanied by removal of the hardcoded values it replaces | Not a runtime issue, but file size and maintainability degrade linearly |
| Regenerating PNGs in every commit during development | Golden baseline test suite (376 tests) runs slow; commit sizes balloon with binary diffs | Batch all rendering changes, regenerate once at end of phase | Immediate -- each golden test compares full RGBA buffers |
| CSS specificity wars from fragmented `<style>` blocks | Page-specific `<style>` rules override style.css rules unpredictably; `!important` proliferation | Consolidate page-specific CSS into style.css with scoped selectors (`.page-fun .card`, etc.) | Gradual -- each new page-specific override adds unpredictability |

## UX Pitfalls

Common user experience mistakes in this domain.

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Compare spinner blocks interaction during geolocation | First-time visitor sees 8+ seconds of spinner before empty state appears | Show empty state with "Add stores" CTA immediately; populate asynchronously when geolocation resolves |
| Button style changes break muscle memory | Users expect the blue button in the same spot to do the same thing; changing padding/size/position during "polish" confuses | Preserve button positions and approximate sizes; only change internal styling (color consistency, border-radius) |
| Rarity badge color changes confuse returning users | "Rare" was pink, now it is purple -- did the meaning change? | If unifying rarity colors, pick colors that communicate meaning (warm=rare, cool=common) rather than matching a palette arbitrarily |
| Removing inline styles from compare.html CTA | The "Set it up" link in compare.html has extensive inline styles (`display:inline-block;padding:0.5rem 1.25rem;background:#005696;color:white;border-radius:0.5rem;text-decoration:none;font-weight:600`); removing these without adding equivalent CSS class leaves an unstyled text link | Replace inline styles with `.btn-primary` class AND verify the class definition in style.css covers all the properties |
| Cone quality upgrade changes familiar visual identity | Users recognize their flavor by the current cone image; dramatic rendering changes make the app feel different | If upgrading topping distribution/coherence, keep the overall silhouette and color palette recognizable -- refine, do not redesign |

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Token rename breaks rendering | LOW | `git diff` on `:root` block to find renames; restore old names, add new names as aliases |
| CSS class rename misses JS references | LOW | `grep -r "old-class-name" docs/` to find remaining references; update in follow-up commit |
| Page `<style>` overrides persist after unification | LOW | `grep -r '<style' docs/*.html` to find all page-scoped blocks; reconcile or remove |
| Partial golden regeneration fails CI | LOW | Run `UPDATE_GOLDENS=1 npx vitest run golden-baselines.test.js` to regenerate all tiers; commit the full set |
| Cone changes interleaved with CSS work | MEDIUM | Revert cone commit, rebase CSS work, re-apply cone changes on clean state, regenerate all baselines |
| Compare double-render from race condition | LOW | Add `{ once: true }` to `sharednav:storechange` event listener, or track resolution state with a boolean flag |
| Rarity unification breaks map popups | LOW | Add context-specific token overrides in `.leaflet-popup-content { --rarity-rare: ...; }` |
| SW cache not bumped | LOW | Increment `CACHE_VERSION` in sw.js, deploy; returning users get fresh assets on next visit |
| JS inline styles missed during audit | LOW | `grep -r 'style\.' docs/*.js` (ignore `style.display`) to find remaining hardcoded color mutations |
| border-radius tokens only partially applied | LOW | `grep 'border-radius:' docs/style.css` to find remaining hardcoded values; convert in follow-up |
